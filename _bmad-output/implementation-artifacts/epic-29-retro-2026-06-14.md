### 做得好的地方

#### W1：在不破坏既有 Agent 语义的前提下加入兼容性

Story 29.1 保持 `createAgentTool()` 不变，并通过共享 launcher helper 新增 `createTaskTool()`。Swift 类型命名约束得到遵守：没有引入名为 `Task` 的 Swift 类型。

Winston（架构师）: "这是正确的兼容形状：保留 SDK canonical 工具，加入 Claude Code alias，并共享执行语义。"

#### W2：在正确层级补上 runtime 缺口

Story 29.2 通过 `SubAgentLauncherNames` 把 launcher-name 检测和 child-tool 过滤放到 Core 层。这避免了两类失败模式：

- 只注册 `Task` 时没有注入 spawner
- 未过滤的 `Agent` 或 `Task` launcher 导致递归 child spawning

实现通过保守默认行为保留了后续递归 opt-in 的设计空间。

#### W3：Skill package context 遵守 progressive disclosure

Story 29.3 使用 `Skill.baseDir` 和 `Skill.supportingFiles` 告诉 agent supporting file 的位置，但不内联文件内容。对 workflow skill 来说，这是合适的平衡：有足够上下文解析路径，不让 prompt 膨胀，也不进行隐式文件读取。

Paige（技术写作）: "这也让 skill 契约更容易解释：路径可发现，内容仍按需读取。"

#### W4：Tool declaration 从有损 enum 限制升级为 lossless 模型

Stories 29.4 和 29.5 把 `allowed-tools` 从有损的 enum-only 路径升级为更丰富的声明模型。实现保留了 SDK 名称、MCP 名称、自定义名称、pattern、unknown declaration 和 diagnostics。

关键设计选择得到了保持：

- unknown declaration 不会静默变成 unrestricted access
- MCP 名称保持 namespace
- `Task` 被识别为 Claude Code-compatible SDK declaration，但没有新增 `ToolRestriction.task` enum case
- pattern declaration 会被解析并报告，但不会被夸大为已强制执行

#### W5：Deferred runtime 字段变得可见

Story 29.6 在保持 protocol signature 稳定的同时，通过 `SubAgentResult.fieldDiagnostics` 暴露 deferred field。SDK 现在会告诉调用方：哪些 schema 接受的字段被忽略，或只被部分兑现。

Murat（QA）: "这是实质性的质量提升。静默 no-op 字段很难调试；diagnostic 是可测试的。"

#### W6：测试计数纪律明显改善

Epic 28 曾指出测试计数反复漂移。Epic 29 的 story 记录持续捕获了全量测试总数，包括 29.4 和 29.5 中 review 修复导致的计数变化。收尾 story 以 5795 项测试通过关闭；发布前修复和回归后，最终记录为 5797 项 XCTest 与 12 项 Swift Testing 通过。

### 挑战与摩擦

#### C1：兼容性工作需要跨层协调

该 epic 触及 `Tools/Advanced`、`Core`、`Types`、`Skills`、`Documentation.docc`、cookbook 文档和测试基础设施。Story 能顺利完成，是因为依赖顺序明确；但相比单模块功能，这类工作有更多边界风险。

Amelia（开发者）: "风险不在某个困难算法，而在跨 tool creation、spawner injection、skill execution、declaration parsing、filtering、diagnostics 和 docs 保持语义一致。"

#### C2：Parser 与 filter 边界情况需要 review 压力

Stories 29.4 和 29.5 都需要 review 推动的修复：

- PascalCase Claude Code 名称最初无法匹配 camelCase 的 `ToolRestriction` raw value。
- 空括号或 malformed 括号 pattern 需要更严格处理。
- MCP normalization 与 mixed-case declaration 需要保持一致。
- 预设 allowlist 曾可能导致 double-filter 并丢弃 MCP 工具，直到 review 修复。

这些问题处理得当，但也说明 parser/filter story 默认就需要 adversarial tests。

#### C3：测试基础设施仍有尖锐边界

Story 29.3 的 progressive-disclosure 测试一开始存在虚假的信心，直到 review 写入一个带唯一 token 的真实 supporting file 才变成有效断言。它还暴露了 request-body 断言中的 JSON escape 问题。共享的 `extractPromptTextFromRequestBody(_:)` helper 是正确修正。

#### C4：独立 E2E 覆盖仍受真实环境约束

Story 29.7 正确拒绝创建 mock-based E2E。发布前已经使用当前项目 `.env` 对 47 个 examples 做真实环境回归并全部通过；但独立的 Task-alias E2E target 仍未落地。Unit、integration 与 examples 覆盖很强；专门的真实 E2E 仍是就绪项。

#### C5：既有 observability 债务仍在本 epic 范围之外

Epic 29 没有解决 subagent 的 EventBus inheritance，也没有解决 Epic 28 留下的 dual step-index concern。这不是 Epic 29 范围失败，但如果后续工作触及 multi-agent observability，它仍然重要。
