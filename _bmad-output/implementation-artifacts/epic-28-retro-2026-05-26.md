## What Went Well

### W1: EventBusBridge Actor Is a Clean Abstraction

The `EventBusBridge` actor (66 lines) perfectly encapsulates the EventBus → SSE pipeline:
- Subscribes to EventBus
- Maps events via stateless `AgentEventSSEMapping.map()`
- Maintains stepIndex counter
- Auto-completes on terminal events

This is the ideal single-responsibility actor — no external dependencies beyond the injected `EventBus` and `EventBroadcaster`. The separation from `executeRun` made both components simpler.

### W2: Per-Call EventBus Injection via Agent.stream() Overload

Story 28.2 added `stream(_ text: String, eventBus: EventBus? = nil)` to Agent, allowing per-call eventBus override without mutating shared agent options. This is the correct approach for concurrent safety — no locks, no temporary mutation, no race conditions. It follows the interface-first pattern at the method level.

### W3: Stateless Mapping Enum Pattern Validated Again

`AgentEventSSEMapping` (56 lines) follows the same `enum` with static function pattern as `TraceEventMapping`. This pattern was proven in Epic 24 and continues to work well — pure function, no state, trivially testable. The deviation was reviewed and accepted. It simplifies the codebase.

## What Could Be Improved

### C1: Story 28.2 Review Found 2 HIGH-Severity Issues

The code review for Story 28.2 found:
1. **H1: Double `broadcaster.complete()` on error path** — bridge calls it on terminal events AND executeRun called it in `!sawResult` fallback. Fixed by removing the executeRun fallback.
2. **H2: Dual stepIndex counters** — executeRun and bridge both increment stepIndex. They stay in sync because both respond to the same ToolCompletedEvent, but it's a design concern.

H1 was a real bug that would have caused issues on error paths. The root cause is that the bridge lifecycle and executeRun lifecycle overlap on terminal event handling.

### C2: Test Count Inconsistencies Continue (6th Epic)

This is now a 6-epic carry-over. The root cause is well-documented: dev agent writes counts during implementation, counts change during review, story file isn't always updated.

### C3: Dual stepIndex Counter Is Fragile

Both `executeRun` (for tracker state) and `EventBusBridge` (for SSE events) maintain independent stepIndex counters. They increment on the same event (ToolCompletedEvent), so they stay synchronized. But if either counter's logic changes independently, they'll diverge silently.

**Recommendation:** Consider having the bridge expose its stepIndex, or use a shared counter source.

### C4: No E2E Test Coverage for SSE Bridge

Story 28.2 modifies `AgentHTTPServer.executeRun` — the core HTTP API entry point — but relies only on unit tests for the bridge. There's no E2E test that exercises the full pipeline (HTTP request → Agent execution → EventBus → Bridge → SSE response). The existing E2E tests for the HTTP server use the previous manual emit path.

## Lessons Learned

### L1: Bridge Pattern Needs Explicit Lifecycle Ownership

Story 28.2's H1 bug (double complete) shows that when a bridge and its host both respond to the same terminal events, ownership must be explicit. The fix was to let the bridge own `broadcaster.complete()` exclusively. The lesson: **one owner per lifecycle** — if a bridge handles terminal events, the host must not also handle them.

### L2: Intentional AC Deviations Should Be Documented at Implementation Time

Story 28.2's AC5 deviation was the right call, but it wasn't flagged until code review. Dev agents should document intentional spec deviations in the story file's completion notes, not wait for review to catch them.

### L3: Per-Call Parameter Injection Is Safer Than Shared State Mutation

The `stream(_:eventBus:)` overload pattern is the right approach for concurrent scenarios. It avoids the entire class of data race bugs that would come from mutating `agent.options.eventBus`. This should be the default pattern for any per-request override in the agent.

### L4: Stateless Mapping Functions Scale Well

The `AgentEventSSEMapping` enum pattern (no state, pure function) has now been validated across 2 epics. It composes cleanly with the bridge pattern and is trivially testable. This should be the template for any future mapping/transformation layer.
